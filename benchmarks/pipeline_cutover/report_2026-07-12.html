Accuracy: perfect. Every one of the 19 real production markets (US, CN, JP, KR, TW, CA, AU, HK, UK, EU, SG, SE, DE, SA, BR, CH, ZA, FI, DK) produced an exact match between the Python original and the C++ port — same scored-ticker count, same rounded average Momentum score, same GGG/GBB/etc. classification breakdown, right down to the per-code counts. Zero mismatches.
Timing: C++ came out slower end-to-end for this exact usage pattern — and here's why. Measured as fresh CLI invocations (one process per --analyze MARKET call, matching how these tools are actually run), bt_pipeline_cli took ~40% longer overall. Isolating the cause: a bare bt_pipeline_cli invocation that does zero DuckDB work (just prints usage and exits) already takes 0.32s, vs Python's 0.03s interpreter start + 0.16s for import duckdb. That gap (~0.32s vs ~0.16s of pure fixed cost) accounts for nearly the entire difference — once it's subtracted out, the actual per-market query work is comparable on both sides (C++ 0.29s vs Python 0.26s of real work per call). The statically-linked DuckDB C++ binary pays a fixed process-startup tax (library static initialization) on every single invocation regardless of work done; Python's per-call cost is lower here only because its interpreter+import overhead happens to be smaller than DuckDB's C++ static-init cost for this specific build. This is a property of the "one process per CLI call" usage pattern, not evidence that the ported logic itself is slower — the same DuckDB engine executes both sides' SQL.
What this implies going forward: for a long-running process (like bt_serve_api, which already showed real wins on live HTTP routes during the cutover), this startup tax is paid once and amortized over many requests — a non-issue. For a CLI invoked repeatedly per-market like this benchmark, the current design pays it on every call. A batch mode (all markets in one process) or a persistent daemon would eliminate this and likely show the C++ engine's real advantage on the query work itself.

What was fixed to make this benchmark possible

While setting this up, bt_pipeline_cli's --process/--analyze stages turned out to never populate WarehousePaths from any CLI flag — they always built against an empty path set, so the shipped tool couldn't actually ingest real data at all. Added --ohlc-dir (repeatable), --companies-parquet, --fundamentals-db, --dvm-global-db, --dvm-composite-db, --viability-db flags (commit 7519b73), verified against real data, 207/207 tests still passing, then used the fixed CLI for this benchmark.

Data source: real production files — 19-market OHLC parquet set (~/market-data-artifacts/seed_ohlc), dvm_global.db (30,785 rows total), dvm_composite.db, fundamentals_cache.db from the real ~/BazaarTalks deployment. Both stacks built independent scratch DuckDB files from the same source data; no shared state between runs.
